# レシート買い物サポートアプリ ― 検証用プロトタイプ 仕様書

## 0. この仕様書の使い方（Claude Codeへ）

このファイルは、第三者にアプリ案を見せてフィードバックをもらうための **検証用プロトタイプ** の仕様です。
「実際に動く製品（MVP）」ではありません。OCR・通知・サーバーなどの重い機能は **すべて"それっぽく見せるだけ"のフェイク** で実装してください。
目的は「このアプリに需要があるか／方向性が正しいか」をテスターに体感してもらうことです。

---

## 1. 目的と検証したいこと

### 目的
「レシートを撮るだけで記録完了し、買うべきものを教えてくれる」という体験を、テスターに触ってもらって反応を得る。

### このプロトタイプで検証したい3点
1. **課題への共感** … 「買い物の段取り・ストック管理が苦手」という課題に共感してもらえるか
2. **撮るだけの手軽さ** … 「レシートを撮るだけ」という入力の軽さに価値を感じてもらえるか
3. **通知の価値** … 「あと◯日で切れそう」と教えてくれることが役立つと思ってもらえるか

> UI/UXの作り込みは今回の検証対象外。見た目は「スマホで見やすい」程度でよく、過度に凝らない。

---

## 2. 技術仕様

- **構成**: フロントエンドのみ。バックエンド・DB・ログインなし。
- **言語/技術**: HTML + CSS + バニラJavaScript（フレームワーク不要）。
- **ファイル構成（例）**:
  - `index.html`（リポジトリ直下）
  - `css/style.css`
  - `js/app.js`
  - `assets/`（レシート画像などを置く場合）
- **画面遷移**: 1ページ内で複数セクションをJSで切り替えるSPA形式（`display`の出し分け）。
- **データ**: JS内のオブジェクト配列で保持（メモリ）。**リロードで初期化されてOK**。`localStorage`は任意（無くてよい）。
- **表示**: スマホ幅を基準にしたレスポンシブ。画面中央に最大幅 430px 程度のモバイル枠を置く形でも可。
- **デプロイ先**: GitHub Pages（リポジトリ直下の `index.html` を公開）。

---

## 3. 画面構成（全5画面）

遷移: 画面0 → 画面1 → 画面2 → 画面3 → 画面4 →（ホームに反映）→ 画面1

### 画面0：オープニング　【検証①課題共感】
- 中央にコンセプトのキャッチコピー
  - 見出し例：「レシートを撮るだけ。あとは考えなくていい。」
  - サブ例：「買い物の段取りやストック管理が苦手な人のための、買い物サポートアプリ」
- ボタン「使ってみる」→ 画面1へ。
- "あるある"と思わせる入り口。長くしない（1画面で完結）。

### 画面1：ホーム　【検証③通知の価値】★最重要
- **起動直後から、すでにデータが埋まっている状態**にする（テスターが最初に「便利そう」と感じる画面）。
- 上部に **通知バナー**：一番期限が近い商品を表示
  - 例：「🔔 トイレットペーパー、あと2日で切れそう」
- **「買うべきもの」リスト**：期限が近い順に商品カードを並べる。各カードに以下を表示
  - 商品名 / あと◯日 / 前回買った店
  - 「あと◯日」が3日以内のものは色で強調（注意を引く）
- 下部に大きめのボタン「レシートを撮る」→ 画面2へ。

### 画面2：撮影　【検証②手軽さ】
- 「撮影する」ボタンを中央に配置（プリセットのレシートを選ぶ簡易UIでも可）。
- **実際のカメラは起動しない。** ボタンを押すと「読み取り中…」のローディング演出（約1〜1.5秒）→ 画面3へ自動遷移。

### 画面3：読み取り結果　【検証②手軽さ】
- 上部に「自動で読み取れた風」の情報：**日付・店名**。
- 商品リスト（各行にチェックボックス、デフォルトは全選択）。
  - 商品名・金額を表示。任意で「修正できる風」のUIを添えてもよい（手入力の保険を見せる）。
- ボタン「これらを記録する」→ 購入履歴に追加し、画面4へ。

### 画面4：サイクル設定
- 記録した商品から「消耗品として管理するもの」を選ぶ。
- 選んだ商品に「◯日で使い切る」を設定（スライダー、またはプリセット選択：7 / 14 / 30 / 60日）。
- ボタン「設定して完了」→ 画面1へ戻る。
- 戻ったホームでは、設定した商品が `今日 + サイクル日数` で計算された「あと◯日」として **リストに反映** されている。

## 4. フェイクデータ仕様

### ホームに最初から登録済みの商品
「あと◯日」は、起動時に `今日 + N日` で `nextDate` を計算し、表示時に `nextDate − 今日` で出す（いつ実行しても正しく表示されるように）。

| 商品名 | 前回の店 | 消費サイクル | 初期の「あと◯日」 |
|---|---|---|---|
| トイレットペーパー | マルエツ | 30日 | あと2日 |
| 食器用洗剤 | マルエツ | 60日 | あと5日 |
| シャンプー | ウエルシア | 45日 | あと12日 |

### スキャン用プリセットレシート（画面2→3で表示）
- 店名：マルエツ ◯◯店
- 日付：今日の日付
- 商品：
  - トイレットペーパー 12ロール … ¥398
  - 食器用洗剤 … ¥298
  - 牛乳 … ¥208
  - 卵 … ¥228
  - 食パン … ¥158

> 商品名・店名は、テスターの生活に近い一般的なものにしてある。必要なら自由に調整可。

## 5. スコープ外（作り込まないこと）

検証用なので、以下は **実装しない**（フェイク／省略）：
- 実際のOCR・文字認識
- 実際のカメラ起動
- 実際のプッシュ通知・バックグラウンドジョブ・定期実行
- サーバー / データベース / ログイン / 認証
- 消費サイクルの自動学習・平均計算（本実装のSTEP4以降の機能）
- 最安値の店の提案・店舗ナビ機能

---

## 6. デプロイ手順（GitHub Pages）

1. GitHubでリポジトリを作成（例：`receipt-shopping-prototype`）。
2. `index.html` をリポジトリ直下に置いてプッシュ。
3. リポジトリの Settings → Pages → Source を `main` ブランチ／`/(root)` に設定。
4. 数十秒後、`https://<ユーザー名>.github.io/<リポジトリ名>/` で公開される。
5. スマホのブラウザで開いて動作確認（キャッシュが残る場合はハードリフレッシュ）。

---

## 7. 実装の進め方（推奨）

1. まず画面遷移の骨組み（5セクションの出し分け）を作る。
2. 画面1ホームを、フェイクデータが入った状態で先に完成させる（一番大事な画面）。
3. 画面2→3の「撮影→読み取り結果」の演出をつなぐ。
4. 画面4のサイクル設定と、ホームへの反映を実装する。
5. 画面0オープニングを足して全体をつなぐ。
6. スマホ表示の最終調整 → GitHub Pagesへデプロイ。
